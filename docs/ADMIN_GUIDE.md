# Робота з адмінкою — як формується сторінка і звідки береться контент

Цей документ пояснює **простими словами**, як влаштована адмінка сайту, як
збирається кожна сторінка та звідки на неї потрапляє контент. Читати можна
без технічних знань — розділи для розробників позначено окремо.

---

## 1. Загальна картина за 30 секунд

Сайт складається з двох частин:

- **Адмінка (Payload CMS)** — панель за адресою **`/admin`**, де редактори
  керують усім контентом.
- **Сайт (Next.js)** — те, що бачить відвідувач.

Головна ідея: **редактор не верстає сторінки, а збирає їх із готових
«цеглинок» (блоків)**. Кожна сторінка — це набір блоків у певному порядку.
Редактор може блок додати, прибрати, переставити місцями або тимчасово
сховати. Сайт автоматично малює те, що зібрано в адмінці.

```
Відвідувач → Сторінка сайту → бере дані з Адмінки → малює блоки по черзі
```

Зміни в адмінці з'являються на сайті **автоматично** одразу після
збереження — нічого перезапускати не треба (див. розділ 7).

---

## 2. Два джерела контенту: Глобали і Колекції

Увесь контент в адмінці зберігається у двох видах сутностей.

### Глобали (Globals) — «сторінки та налаштування, які існують в однині»

Глобал — це один-єдиний запис. Наприклад, Головна сторінка одна, тож її
контент — це глобал. У меню адмінки вони згруповані окремо.

| Глобал | Що це | Керує сторінкою |
|---|---|---|
| **Home Page** | Головна сторінка | `/` |
| **About Page** | Сторінка «Про нас» | `/about` |
| **Contact Page** | Сторінка контактів | `/contact` |
| **Signals Page** | Сторінка сигналів | `/signals` |
| **Live Feed Page** | Стрічка подій | `/live-feed` |
| **News Page** | Сторінка новин (обгортка списку) | `/news` |
| **Header** | Верхнє меню/шапка | усі сторінки |
| **Footer** | Підвал сайту | усі сторінки |
| **Site Settings** | Загальні налаштування: SEO, аналітика, бічні промо-блоки для всього сайту | усі сторінки |
| **Translations** | Переклади (якщо ввімкнено кілька мов) | усі сторінки |

### Колекції (Collections) — «списки записів»

Колекція — це список однотипних записів (як таблиця). Наприклад, новинних
статей багато, тож кожна стаття — це запис у колекції **News**.

| Колекція | Що зберігає |
|---|---|
| **News** | Новинні статті |
| **Signals** | Сигнали |
| **Live Feed** | Записи стрічки подій |
| **Case Studies** | Кейси |
| **Categories** | Категорії |
| **Tags** | Теги |
| **Media** | Завантажені картинки/файли |
| **Contact Submissions** | Заявки, надіслані через форму контактів |
| **Users** | Користувачі адмінки (редактори/адміни) |

**Ключова різниця:**
- Контент **глобала** редактор пише вручну (заголовки, тексти, промо-картки).
- Контент **колекції** — це окремі записи, які потім **автоматично
  підтягуються** у відповідні блоки на сторінках (наприклад, останні новинні
  статті або останні записи стрічки).

## 3. Як улаштована сторінка: Регіони → Блоки

Кожна сторінка-глобал усередині поділена на **регіони** (в адмінці це вкладки
угорі форми). Типово це «Main content» (основна колонка) та «Sidebar» (бічна
колонка).

Усередині кожного регіону є поле-**конструктор блоків** — саме тут редактор
збирає сторінку. Натискаючи «Add block», редактор бачить список **дозволених
для цього регіону блоків** і додає потрібний. Блоки можна перетягувати, щоб
змінити порядок.

Що з чого складається конкретна сторінка:

| Сторінка | Регіон (вкладка) | Доступні блоки |
|---|---|---|
| **Home** | Left sidebar | Signal Feed, Summary, Real Card |
| **Home** | Main content | Analysis, Live Feed, Category Section |
| **About** | Content | заголовок + текст + CTA (звичайні поля) |
| **About** | Right sidebar | Live Feed, Real Card |
| **Contact** | Main content | Contact Form Fields |
| **Contact** | Right sidebar | Contact Methods, Contact Value |
| **Signals** | Main content | Signals List |
| **Signals** | Right sidebar | Real Card |
| **Live Feed** | Main content | Live Feed List |
| **Live Feed** | Right sidebar | Real Card |
| **News** | Main content | News List |
| **News** | Right sidebar | Summary |

> Набір блоків у кожному регіоні навмисно обмежений — так редактор не
> поставить блок туди, де він візуально не підходить.

Крім цих регіонів, у **Site Settings → Sidebar** є два поля (`promoBlocks`,
`sponsoredBlocks`) — промо-блоки, які виводяться в бічних колонках **на всьому
сайті**, а не лише на одній сторінці.

## 4. Що таке блок і які вони бувають

**Блок** — це самостійний шматок сторінки з власним набором полів. Наприклад,
блок «Summary» має поле «Tabs» зі списком пунктів; блок «Real Card» — це
промо-картка з картинкою і кнопкою.

Блоки бувають двох типів за джерелом контенту:

### А) Блоки з ручним контентом
Редактор сам заповнює всі поля блоку. Що написав — те й показується.
Приклади: **Summary**, **Real Card**, **Analysis**, **Contact Methods**.

### Б) Блоки, що тягнуть дані з колекцій
Редактор задає лише **налаштування** (заголовок, підзаголовок, скільки
записів показати — `limit`), а самі елементи автоматично беруться з колекції.
Приклади:
- **News List** → показує статті з колекції **News**
- **Live Feed List** → показує записи з колекції **Live Feed**
- **Signals List** → показує сигнали з колекції **Signals**
- **Category Section** → показує статті певної категорії

Тобто щоб змінити список новинних статей, редактор **не чіпає блок** — він
додає/редагує записи в колекції News, і блок оновлюється сам.

### Спеціальна опція «Hide on site»
Майже кожен блок має чекбокс **«Hide on site»**. Він тимчасово ховає блок зі
сторінки, **не видаляючи** його. Зручно, коли треба прибрати щось на час, але
не втратити налаштування.

### Групування однакових блоків (важлива деталь)
Два блоки виводяться особливо: якщо поставити кілька **Signal Feed** підряд —
вони об'єднуються в один спільний віджет із заголовком. Так само кілька
**Category Section** підряд збираються в один блок із перемикачем категорій.
Порядок при цьому зберігається.

## 5. Покроково: як відредагувати сторінку

1. Зайти в адмінку: **`/admin`**, увійти під своїм акаунтом.
2. У меню зліва вибрати потрібний глобал (наприклад, **Home Page**) або
   колекцію (наприклад, **News**).
3. Для сторінки-глобала — вибрати вкладку регіону (наприклад, **Main
   content**).
4. У конструкторі блоків:
   - **Додати блок** — кнопка «Add block» → вибрати тип.
   - **Змінити порядок** — перетягнути блок за ручку.
   - **Тимчасово сховати** — увімкнути «Hide on site».
   - **Видалити** — меню блоку (три крапки) → Remove.
5. Заповнити поля блоку.
6. Натиснути **Save**. Зміни з'являться на сайті одразу.

> Щоб змінити список (статті, сигнали, події) — редагуйте відповідну
> **колекцію**, а не блок на сторінці.

---

## 6. Ролі та доступи

У користувача адмінки є роль:

- **Admin** — повний доступ, зокрема керування користувачами (створення нових
  редакторів, зміна ролей).
- **Editor** — може редагувати контент (глобали, статті тощо), але **не**
  керує користувачами.

Створювати нових користувачів і призначати ролі може лише **Admin**.

---

## 7. Чернетки, автозбереження та live-preview

- Колекції на кшталт **News** підтримують **чернетки** (drafts) з
  автозбереженням і **історію версій** (зберігається до 25 версій на запис).
  Тобто можна працювати над чернеткою, не публікуючи її, і за потреби
  відкотитися до попередньої версії.
- Є **Live Preview** — попередній перегляд сторінки поруч із формою
  редагування, щоб бачити зміни ще до публікації.

---

## 8. Чому зміни з'являються миттєво (кеш)

Сайт кешує дані заради швидкості. Але при кожному збереженні чи видаленні в
адмінці автоматично спрацьовує механізм скидання кешу (revalidate), тож
відвідувач одразу бачить актуальний контент. Спеціальних дій від редактора не
потрібно.

## 9. Технічна довідка (для розробників)

Стек: **Payload CMS 3** (адмінка + API) поверх **Next.js App Router**.

### Ключові файли й директорії
- `src/payload.config.ts` — головний конфіг: реєстрація колекцій і глобалів.
- `src/globals/*.ts` — описи глобалів. Регіони — це `type: 'tabs'`, а
  конструктор блоків — поле `type: 'blocks'` з масивом дозволених блоків.
- `src/collections/**` — описи колекцій.
- `src/blocks/<Name>/config.ts` — **схема блоку** для адмінки (його поля).
- `src/blocks/<Name>/Component.tsx` — **рендер блоку** на фронтенді.
- `src/blocks/regionBlockComponents.tsx` — реєстр `blockType → React-компонент`.
- `src/blocks/RenderBlockList.tsx` — універсальний рендер регіону: фільтрує
  сховані блоки (`hidden`), ставить роздільники, підбирає компонент за
  `blockType`.
- `src/blocks/RenderHomeBlocks.tsx` — рендер регіонів головної з групуванням
  `signal-feed` та `category-section` (логіка групування — `homeBlocks.ts`,
  функція `planRegion`).
- `src/blocks/_shared/fields.ts` — спільне поле `hiddenField` («Hide on site»).
- `src/utilities/getPageContent.ts` — завантаження глобалів
  (`getHomePageContent()` тощо), кешоване через `unstable_cache` з тегом
  `payload`.
- `src/hooks/revalidateFrontCache.ts` — хуки `afterChange`/`afterDelete`, що
  викликають `revalidateTag('payload')` після будь-якої зміни контенту.
- `src/access/isAdminOrEditor.ts`, `src/access/isAdmin.ts` — контроль доступу.

### Як додати новий блок
1. Створити `src/blocks/<Name>/config.ts` (експортує `Block` зі `slug` і
   `fields`; за потреби додати `hiddenField`).
2. Створити `src/blocks/<Name>/Component.tsx` (рендер; для даних із колекцій —
   тягнути через утиліти в `src/utilities/`).
3. Зареєструвати компонент у `regionBlockComponents.tsx` під ключем = `slug`.
4. Додати `Block` у масив `blocks: [...]` потрібного регіону в глобалі
   (`src/globals/*.ts`).
5. Оновити типи: `pnpm generate:types`.

### Потік рендерингу (для головної як приклад)
```
GET /  →  src/app/(frontend)/page.tsx
       →  components/Home/Main.tsx
       →  getHomePageContent()            // глобал home-page з Payload (кеш)
       →  SidebarRegion / MainRegion      // RenderHomeBlocks.tsx
       →  planRegion() групує блоки        // homeBlocks.ts
       →  regionBlockComponents[blockType] // потрібний компонент кожного блоку
       →  блоки з колекцій самі роблять запити при рендері
```

Інші сторінки (`about`, `contact`, `signals`, `live-feed`, `news`) працюють за
тим самим шаблоном, але через універсальний `RenderBlockList`.
